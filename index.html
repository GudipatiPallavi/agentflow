UX Design Specification · v1.0 · 2025

AgentFlow

A UX design specification for an AI workflow builder designed for non-technical professionals. Focused on simplicity, trust, transparency, and agentic AI interactions.

Designed by Senior UX, Google
Platform Web / Desktop
Target User Non-technical professionals
AI Paradigm Agentic workflows

01 — Design Principles

Four pillars that govern every decision

AgentFlow's UX is built on four non-negotiable principles. Every interaction, every screen, every micro-copy choice is evaluated against these.

Simplicity Trust Transparency Human Control
🧭
Simplicity
Non-technical users should never feel lost. Progressive disclosure hides complexity until needed. Plain language over jargon. Templates and guided setup over blank canvases. Every feature must justify its cognitive cost.
🤝
Trust
Trust is earned, not assumed. AgentFlow makes AI actions legible before they happen. Users see what an agent will do, not just what it did. Confidence scores, source citations, and audit trails are default-on, not settings.
🔍
Transparency
The system explains its reasoning at every step. No black boxes. When an agent makes a decision, it shows the data it used. When it's uncertain, it says so. Users can always inspect the "why" behind any AI output.
🎛️
Human Control
Agents assist, they don't replace human judgment on consequential actions. Approval gates are first-class citizens in the UI — not add-ons. Users can pause, modify, or abort any workflow at any point without losing state.

02 — Information Architecture

How the product is structured

AgentFlow organises around five primary areas. The IA is deliberately flat — maximum two levels deep — to reduce navigation overhead for non-technical users.

IA-1 Approvals as a top-level section — not nested under workflows. Non-technical users need to review and approve AI actions regularly. Burying approvals creates missed decisions and erodes trust.
IA-2 Flat hierarchy intentional — users should reach any screen in 2 clicks maximum. Depth is an enemy of adoption for non-technical audiences.

Node Types

When a user adds a trigger, the canvas suggests the next 3 most common following steps based on workflow patterns. Reduces blank canvas anxiety for non-technical users.

06 — Human Approval Screens

The moment that builds trust

Approval screens are the most trust-critical surface in AgentFlow. They must give enough context to make a confident decision, without overwhelming the reviewer with technical details.

⏸️
Agent is waiting for your decision
Customer Onboarding Sequence · Step 2 of 5 · Paused 4 minutes ago
What the agent wants to do
Send a personalised welcome email to Sarah Chen at Acme Corp
Why it's doing this
Sarah signed up 12 minutes ago. The enrichment step identified her as a VP-level contact at a 200+ person company — your workflow rules flag this tier for human review before first contact.
What it will send
Subject: Welcome to AgentFlow, Sarah

Hi Sarah, great to have you on board. Given your role at Acme Corp, I'd love to schedule a 20-min call...
Confidence & Sources
95% contact match confidence Sources: LinkedIn, Clearbit, CRM
A-1 "What the agent wants to do" — one clear sentence in plain English before any detail. Reviewers must instantly understand the action without reading context first.
A-2 Confidence scores are surfaced, not hidden in settings. A 95% match score gives reviewers permission to approve quickly. A 60% score signals caution. Transparency earns trust.
A-3 Three decision options, not two — Approve / Modify / Reject. "Modify before sending" is the critical middle path that US-only Approve/Reject designs miss. It respects the user's judgment without blocking the workflow entirely.

M-1 Logs in plain English, not code — "Agent is enriching contact data" not "POST /api/enrichment 200 OK". Non-technical users must read logs and feel informed, not confused.
M-2 Approval pauses are surfaced in the log with a ⏸ icon and a clear reason. Users see exactly why the workflow stopped and what they need to do — reducing support tickets about "stuck workflows."

08 — Error Handling

Failures that don't feel like failures

Error states are where trust is lost — or saved. AgentFlow's error handling follows one rule: always tell the user what happened, why, and exactly what they can do next.

🔌
Integration Error
External service is unreachable. Agent pauses automatically, preserves state, and notifies the user. Retry is one click. No data is lost.
🤷
Low Confidence
Agent is uncertain about a decision. Instead of guessing, it routes to human review with the uncertainty explained. Treated as an approval, not an error.
⏱️
Timeout / Stall
Workflow hasn't progressed in X minutes. User is notified with context — last successful step, what the agent was attempting, and suggested actions.
⚠️
Step failed: CRM Update
The agent tried to update Salesforce with Sarah's enriched contact data, but the connection timed out. Your workflow is paused — no data was lost and no duplicate records were created.

This usually happens when Salesforce is under maintenance. You can retry now or skip this step and update manually later.
E-1 "No data was lost" — this phrase must appear in every error message where state is preserved. It's the single most anxiety-reducing statement for non-technical users. Say it explicitly.
E-2 Never show raw error codes to non-technical users. "Salesforce connection timed out" is acceptable. "Error 504: Gateway Timeout on POST /services/data/v57.0/sobjects/Contact" is not.
E-3 Low-confidence states are not errors — they are transparency signals. Routing uncertainty to human review is a feature, not a failure. The UI distinguishes the two clearly with different visual treatment (amber vs red).

09 — Component Spec

Design system decisions

Key component decisions that enforce the four design principles across every screen.

Component Design Decision Principle
Status Indicator Colour + icon + text label — never colour alone. Accessible for colourblind users. Running (blue pulse), Paused (amber), Done (green), Failed (red). Transparency
Approval Gate Node Visually distinct amber border on canvas. Cannot be accidentally deleted — requires explicit "Remove gate" confirmation. Always shows assignee and timeout. Human Control
Confidence Score Shown as percentage + colour bar. Green >85%, Amber 60–85%, Red <60%. Red automatically routes to approval. Never hidden in settings. Transparency, Trust
Error Message Three-part structure: what happened (plain English) + reassurance (state preserved) + what to do next (1–3 action buttons). Never raw API errors. Simplicity, Trust
AI Explanation "Why did the agent do this?" is available on every AI action node — one click, expandable panel. Never buried in logs. Uses natural language, not JSON. Transparency
Undo / Abort Global keyboard shortcut ⌘Z for canvas. "Pause workflow" button always visible in monitoring header. "Abort run" requires confirmation with consequence summary. Human Control
Empty States Never blank. Always includes: what this section is for + a primary action button + a link to a template. Reduces blank canvas anxiety. Simplicity
Notification Approval requests: push + email + in-app badge. Run completions: in-app only (not email). Failures: push + email. User controls notification level per workflow. Human Control
